## Problem Statement

The Chairman Dashboard has no UI entry point for Stage 0 of the 25-stage venture lifecycle. Stage 0 (venture initiation — sourcing, scoring, routing, nursery) runs entirely through the CLI (`executeStageZero`). The "Explore Opportunities" link on the briefing page is conditional (only appears when no decisions pending), points to a non-existent route (`/chairman/opportunities`), and has no navigation target.

The chairman needs a permanent, always-accessible way to:
1. Browse opportunity blueprints generated by the competitive intelligence pipeline
2. Review scoring signals (capability alignment, portfolio synergy, stage-of-death prediction)
3. Commit: approve a blueprint into Stage 1, park it in the nursery, or reject it
4. Monitor the venture nursery (parked opportunities with reactivation triggers)
5. Trigger the Stage 0 analysis pipeline (asynchronously, via Claude Code)

## Discovery Summary

### Current State
- **Backend**: Stage 0 fully implemented — `executeStageZero()`, entry paths, synthesis, chairman review, venture nursery (park/reactivate), counterfactual engine, stage-of-death predictor, cross-stage data contracts
- **Existing UI components (disconnected from V3)**:
  - `OpportunitySourcingDashboard.jsx` — legacy JSX, queries `opportunity_blueprints`
  - `OpportunityBrowseTab.tsx` — TypeScript, browse/score/select blueprints
  - `ventureIdeationService.ts` — full service layer with `OpportunityBlueprint` types
  - `blueprintScoring.ts`, `blueprintSelection.ts` — assessment and selection services
  - `opportunityToVentureAdapter.ts` — blueprint → venture transform
  - API endpoints: `/api/v2/blueprints/`, `.../review`, `.../instantiate`
- **Claude Code integration**: Clipboard-only (copy markdown context, no remote execution)
- **Navigation**: 5 chairman items + 3 builder items in sidebar; 5 mobile tabs

### Chosen Placement
**Option D: Dual placement** — sidebar nav item (`/chairman/opportunities`) + permanent CTA on the briefing dashboard page. Maximum discoverability without cluttering mobile tabs (briefing CTA serves as mobile entry point).

### Trigger Mechanism
DB work queue pattern: UI writes a request record → Claude Code picks it up asynchronously → status indicator on the page shows pending/in-progress/complete. The `selectBlueprint()` function (which creates the venture record) runs *after* Stage 0 analysis completes.

## Analysis

### Arguments For
- **The pipeline entry point doesn't exist in UI** — no way to create a venture from the web app that respects Stage 0
- **90% of the backend is built** — blueprint APIs, scoring, selection, adapter, nursery are all implemented but disconnected
- **High feasibility** — estimated 2 focused days for MVP; it's a wiring job, not a construction job
- **Compounds over time** — every blueprint selection generates structured data; within months, a venture pattern library with outcome data
- **Nursery becomes visible** — parked ventures get a visual home, preventing "forgot I parked that"
- **Reference implementation** — the interaction pattern for Stage 0 defines the vocabulary for all subsequent stage UIs

### Arguments Against
- **Scope risk** — browse + review + commit + nursery + counterfactual signals is a lot for one page
- **Async trigger UX is tricky** — chairman clicks "Approve" and feedback loop is unclear without careful status display
- **Blueprint pipeline may be empty** — if competitive intelligence hasn't run, page is blank (likely initial state)

## Team Perspectives

### Challenger
- **Blind Spots**:
  1. Stage 0 is a commit decision, not a browse activity — page design must prioritize the commit flow, not discovery aesthetics
  2. Venture nursery (parked ventures) has no visual home — reactivation is orphaned without it
  3. Counterfactual engine and stage-of-death predictor must surface *before* the commit, not after
- **Assumptions at Risk**:
  1. Legacy components may be liabilities rather than assets (different data contracts, state management, visual consistency)
  2. DB work queue fails silently without status indicators and feedback loops
  3. Single-page scope may demand a wizard/step-flow pattern, not a dashboard page
- **Worst Case**: Page becomes a browseable gallery with a clipboard button; chairman bypasses it for CLI; nursery ventures accumulate invisibly

### Visionary
- **Opportunities**:
  1. Permanent Opportunity Funnel — shift from episodic to systematic venture creation with auditable intake
  2. Counterfactual Engine as strategic muscle — pre-mortem tool that runs *before* the venture starts
  3. Venture Nursery as strategic reserve — visible "option book" for deferred opportunities
- **Synergies**:
  - Briefing CTA carries intelligence context into the exploration page ("two competitors just exited this space")
  - `opportunityToVentureAdapter` becomes a versioned founding artifact traceable to the originating blueprint
  - Reference implementation for the entire venture lifecycle UI vocabulary
- **Upside Scenario**: Page evolves into the chairman's primary strategic intelligence surface — pre-scored shortlists, counterfactual context loaded, outcome data from 40-60 blueprints over 12+ months, visible nursery with trigger-based reactivation. Venture investment committee workflow for a single operator.

### Pragmatist
- **Feasibility**: 3/10 difficulty (low) — this is a wiring job, not a construction job
- **Resource Requirements**: 11-15 developer-hours (2 focused days), zero infrastructure spend
- **Constraints**:
  1. Mobile 6th tab problem: solved by NOT adding to mobile tabs (use briefing CTA as mobile entry point, following Prefs exclusion precedent)
  2. Stage 0 backend is CLI-only: work queue with status indicator is correct approach
  3. Blueprint pipeline may be empty initially: need graceful empty state with clear guidance
- **Recommended Path**:
  1. Day 1 AM: Schema verification sprint (Supabase import paths, `sd_phase_handoffs` FK, `opportunity_blueprints` columns)
  2. Day 1 PM: Build MVP page (`ExploreOpportunitiesPage.tsx`) with TanStack Query + blueprint cards + approve/reject actions
  3. Day 1 EOD: Wire route (`chairmanRoutesV3.tsx`) + nav (`chairman-nav-config.ts`) — 30 minutes
  4. Day 2 AM: Briefing CTA in `BriefingDashboard.tsx`, anchored to existing R&D Opportunities section
  5. Day 2 PM: Work queue status display + empty state handling

### Synthesis
- **Consensus Points**: Don't port legacy JSX; rebuild as V3 TypeScript. Nursery must be visible. Stage 0 signals before commit. Don't add 6th mobile tab.
- **Tension Points**: Page vs wizard (resolve: page first, detail view as multi-step). Silent vs explicit async (resolve: work queue + status indicator).
- **Composite Risk**: Medium — infrastructure exists, main risk is scope creep

## Out of Scope
- Building a Claude Code Remote execution bridge (clipboard + work queue for now)
- Blueprint *creation* form (this page consumes existing blueprints, doesn't create them)
- Full Stage 0 wizard with all sub-steps visible (MVP is browse → approve/park/reject)
- Modifications to the competitive intelligence pipeline itself

## Open Questions
1. Should the nursery section be a tab on this page, or a collapsible section below the active blueprints?
2. What happens when the chairman approves a blueprint but Claude Code isn't running — how long should the work queue item persist before showing a warning?
3. Should the counterfactual/stage-of-death predictions be inline on each blueprint card, or revealed on a detail/drill-down view?
4. Should there be a "manual entry" path (chairman types a raw idea) alongside blueprint selection?
