# Vision: Explore Opportunities — Stage Zero UI Surface

## Executive Summary

The Chairman Dashboard currently has no UI entry point for Stage 0 of the 25-stage venture lifecycle. While the backend is fully implemented — entry paths, synthesis, chairman review, venture nursery, counterfactual engine, and stage-of-death predictor — the only way to initiate a new venture is through the CLI. This creates a fundamental gap: the chairman's primary decision-making interface (the web dashboard) cannot perform one of its most important functions (venture creation and routing).

This vision defines the "Explore Opportunities" page as the UI face of Stage 0. It surfaces opportunity blueprints from the competitive intelligence pipeline, provides pre-commit decision signals (scoring, counterfactual analysis, mortality predictions), enables the approve/park/reject routing decision, and makes the venture nursery visible as a strategic reserve. The page is permanently accessible via dual placement: a sidebar navigation item and a briefing page CTA.

Beyond immediate utility, this page establishes the interaction pattern and design vocabulary for all subsequent venture lifecycle stage UIs (Stage 10 brand approval, Stage 22 release gate, Stage 25 portfolio review). It is the reference implementation for how the chairman interacts with venture data throughout the entire lifecycle.

## Problem Statement

The chairman cannot create, review, or route new venture opportunities from the web dashboard. Stage 0 — the venture initiation process — runs entirely through Claude Code CLI (`executeStageZero`). This means:

- **No visual deal flow**: Opportunity blueprints generated by the competitive intelligence pipeline have no consumption surface in the V3 UI
- **Nursery is invisible**: Parked ventures (deferred, not rejected) accumulate in the database with no way to browse, monitor, or reactivate them
- **Decision signals are CLI-only**: Counterfactual analysis and stage-of-death predictions exist but never reach the chairman's primary interface
- **The "Explore Opportunities" link on the briefing page** currently points to a non-existent route (`/chairman/opportunities`) and only appears conditionally when there are no pending decisions

The existing UI components (`OpportunitySourcingDashboard.jsx`, `OpportunityBrowseTab.tsx`) were built for earlier app generations and are not connected to the V3 Chairman Shell.

## Personas

### Chairman (Primary — sole user)
- **Goals**: Maintain strategic oversight of the venture pipeline; make informed approve/park/reject decisions quickly; monitor the strategic option book (nursery)
- **Mindset**: "What's in the pipeline? What should I act on? What did I park that might be ready now?" — executive decision-maker, not a researcher
- **Key activities**: Browse scored blueprints, review pre-commit signals, route opportunities (approve → Stage 1 / park → nursery / reject), monitor nursery for reactivation triggers
- **Constraints**: Time-limited; wants high-signal, low-noise interactions; uses the app on both desktop and mobile

### EVA (System — autonomous intelligence)
- **Goals**: Surface the best-scored opportunities with decision-quality context; trigger reactivation alerts for nursery ventures when timing conditions change
- **Mindset**: Proactive advisor that pre-scores, pre-filters, and pre-analyzes so the chairman makes better decisions faster
- **Key activities**: Generate and score opportunity blueprints, run counterfactual analysis, compute stage-of-death predictions, monitor nursery trigger conditions

## Key Decision Points

### Blueprint Routing Decision (the primary action)
The page's highest-value interaction. For each opportunity blueprint, the chairman chooses:
- **Approve** → Writes a work queue record for Claude Code Stage 0 execution → blueprint status becomes `approved` → venture appears in `/chairman/ventures` after completion
- **Park as Nursery** → Moves to nursery with a review trigger (30d or 90d) → visible in nursery section
- **Reject** → Blueprint status becomes `rejected` with optional rationale → archived

### Pre-Commit Signal Review
Before making the routing decision, the chairman should see:
- **Opportunity score** (composite from competitive intelligence)
- **Capability alignment** (how well does this fit existing capabilities?)
- **Portfolio synergy** (how does this complement active ventures?)
- **Stage-of-death prediction** (where is this most likely to fail in the 25-stage pipeline?)
- **Counterfactual summary** (what would need to be different for this to succeed?)

These signals must appear *before* the commit, not after — they are decision inputs, not post-hoc analytics.

### Nursery Reactivation
Parked ventures resurface when their trigger conditions are met (time-based review interval, or market signal match). The chairman can then re-evaluate and route them.

## Integration Patterns

### Claude Code Async Bridge (DB Work Queue)
The UI cannot call `executeStageZero` directly (CLI-only). Integration uses a shared database as the communication bus:
1. UI writes to `stage_zero_requests` table: `{blueprint_id, status: 'pending', requested_at}`
2. Claude Code polls or receives the request via `sd:next` / work queue check
3. Claude Code runs Stage 0, updates status to `in_progress` then `completed`
4. UI shows real-time status via TanStack Query polling: "Stage 0 in progress for [Blueprint Title]"
5. On completion, venture appears in `/chairman/ventures`

### Existing Service Layer (direct consumption)
- `ventureIdeationService.getOpportunityBlueprints()` — blueprint CRUD
- `blueprintScoring.batchAssessBlueprints()` — scoring with cache
- `blueprintSelection.selectBlueprint()` — creates venture from blueprint
- `opportunityToVentureAdapter.transformBlueprint()` — data transformation

### Briefing Page Integration
The briefing CTA can carry intelligence context (e.g., "NicheBrief AI competitor just exited this market segment") as URL params or shared state, providing "why now" context when the chairman navigates to the opportunities page.

## Evolution Plan

### Phase 1: MVP (2 days)
- Blueprint card list with scores and approve/park/reject actions
- Route + sidebar nav + briefing CTA wiring
- Work queue write + status indicator
- Empty state with guidance ("No blueprints available — run the opportunity generation pipeline")

### Phase 2: Nursery Visibility (1 day)
- Nursery section below active blueprints (parked ventures with trigger dates)
- Reactivation action (move back to active review)
- Trigger countdown/status display

### Phase 3: Decision Signals (2-3 days)
- Blueprint detail view (`/chairman/opportunities/:id`) with full scoring breakdown
- Stage-of-death prediction visualization (bar chart or mortality curve)
- Counterfactual summary panel
- Comparison view (side-by-side two blueprints)

### Phase 4: Full Stage 0 Wizard (future)
- Multi-step flow: browse → review signals → customize parameters → commit
- Manual entry path (raw idea alongside blueprint selection)
- Stage 0 result ingestion (callback that marks work queue item complete)

## Out of Scope

- **Claude Code Remote execution bridge** — clipboard + work queue for now; true remote execution is a separate infrastructure effort
- **Blueprint creation/generation** — this page consumes existing blueprints; the competitive intelligence pipeline that generates them is a separate system
- **Modifications to the Stage 0 backend** — the CLI-side `executeStageZero` is not changing; only the UI trigger mechanism is new
- **Other stage UIs** (Stage 10, 22, 25) — this page establishes the pattern; subsequent stage UIs are separate SDs
- **Mobile tab bar redesign** — the 5-item mobile tab bar is not changing; briefing CTA is the mobile entry point

## Success Criteria

1. **Blueprint consumption**: Chairman can browse, filter, and act on opportunity blueprints from the web UI within 30 days of shipping
2. **Venture initiation**: At least 1 venture is initiated through the UI (approve → work queue → Stage 0 → venture created) within 30 days
3. **Nursery visibility**: Parked ventures are visible with trigger dates; at least 1 nursery reactivation occurs within 90 days
4. **Sub-2-minute workflow**: Chairman can review a blueprint's scores and make a routing decision in under 2 minutes
5. **Mobile accessibility**: The briefing CTA provides a functional mobile entry point; page is responsive and usable on phone
6. **Reference pattern established**: The interaction pattern (card list → scores → action buttons → status feedback) is reusable for Stage 10/22/25 UIs
7. **Empty state handled**: When no blueprints exist, page shows clear guidance rather than a blank screen
8. **Async feedback loop**: Work queue status is visible; chairman knows whether Stage 0 execution has been picked up, is in progress, or completed
