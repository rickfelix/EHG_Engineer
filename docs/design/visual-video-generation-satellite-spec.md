## §2 — Provider abstraction (the deployInstance-primitive pattern)
One primitive: `generateAsset(capability, spec, constraints) → {asset, provenance, cost}` with **provider-per-capability routing**: visual media → RunwayML primary (per the chairman's ruling; Gen-4.5/Seedance/Aleph + the full catalog incl. audio) | Gemini = live image fallback | future providers register against capabilities, never against call sites. **RECIPES (v1.2):** Runway's 7 pre-composed workflow chains are evaluated BEFORE hand-building any multi-step chain in the primitive — a provider-side recipe that matches a §1 lane (e.g. narrated demo composition) is preferred over local orchestration; recipes route through the same primitive with chain-level provenance. **Upscalers** (image/video) register as a §4 quality-REPAIR rung — a gate-failing asset may be upscaled-and-rejudged once before regeneration. Async-task handling is first-class (Runway's SDK model: task polling + `TaskFailedError` → the failure is a typed outcome, never a silent empty asset — the fail-toward-flattery ban applies). Provider keys are secrets-managed per the hosting standard; **test-mode generates watermarked/sandboxed outputs** so the simulated run and APA fixtures never consume production quota silently.

## §3 — Cost model (spine §3 applied)
Every generation attributes to venture × role × activity against a granted spend envelope; per-asset cost is recorded on the asset's provenance. An **asset library with reuse-before-regenerate** is the default cost control (generation is the fallback, not the reflex). Budget exceedance is a spine-§5 exception, not a silent overrun. Provider pricing is part of the envelope math — video generation is orders costlier than image; the routing layer surfaces cost *before* the task fires for anything above a per-asset threshold.

## §4 — Quality gate (the taste line, generalized to generated assets)
1. **Brand-genome conformance** — machine-checked first (palette/typography/tone tokens).
2. **The L5 judge rules apply verbatim:** generated assets are judged brand-stripped + multimodal (the actual pixels, never the generation prompt's description of them) — a generation judged by its own prompt is the decorative-computation class with a paintbrush.
3. **No fabricated evidence in assets:** a generated image may not synthesize social proof, customer counts, testimonials, or product screenshots of features that don't exist — the D1 trust-defect class applies to pixels exactly as to copy. Machine-screened (text-in-image extraction against the venture's claims registry), not honor-system.
4. **Chairman taste checkpoints** where his preferences are the standard (hero images initially) — graduating to machine-precheck as the calibration set grows, per the APA pattern.

## §5 — Envelope registration (today's lesson, applied at birth)
The capability registers into `v_unified_capabilities` **only with delivery evidence** — a real generated asset, through the real provider path, passing the §4 gate, cost-attributed. Registration-before-evidence is banned (the capability-envelope R6 gate would otherwise pass ventures on a capability that is itself vaporware — the recursion the week's doctrine exists to stop). Until registered: Stage-0 must not select ventures premised on it.

## §6 — Sequencing + acceptance
Build order: §2 primitive + Gemini image lane (smallest, satisfies the standing hero-image ask) → §4 gate → §5 registration → Runway video lane → §1-C/D consumers. **Acceptance is mock-distinguishing:** (a) a stubbed provider returning placeholder assets fails §4.2's multimodal judge and §5's delivery evidence mechanically; (b) a generated hero image for the fresh venture passes brand-genome + taste checkpoint and lands on a real landing page; (c) one demo video composes from real product screenshots through the Runway lane inside its cost envelope. SDs cut only after Adam + coordinator accuracy check, per process.
