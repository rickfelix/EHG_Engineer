# Design proposal — Complexity-tiered worker assignment (effort-matched fleet)

**Originated by:** Chairman (2026-06-26) — "assign work based on complexity/effort required so we don't burn powerful-effort tokens on simple SDs; have three standing workers (Low/Medium/High), each picking up only work suited to its effort tier." **Architected by:** Adam (advisory). **Status:** DRAFT design for coordinator review (the coordinator owns fleet dispatch + claim-eligibility).

## Goal
Match **SD complexity** → **worker effort tier** so the long tail of simple SDs (QFs, docs, small infra) runs on a cheap **Low (Sonnet)** worker, reserving the expensive **High (Opus, high-effort)** worker for risky/complex/decomposed work. This is the Targeting Machine / model-tiering principle ([[feedback-scout-cheap-dig-deep-model-tiering]]) applied to the FLEET, not just within a single agent.

## Architecture (6 components; reuse-heavy)

**1. Complexity rubric → `effort_tier ∈ {low, medium, high}` per SD.** REUSE existing primitives, don't reinvent:
- `scripts/classify-quick-fix.js` `CLASSIFICATION_RULES` (LOC threshold + risk-keyword force-escalate) + the CLAUDE.md Work Item Routing tiers (T1 ≤30 / T2 31-75 / T3 >75, risk-force).
- **LOW**: small est-LOC (≤~50-75), single-concern, `sd_type ∈ {docs, chore, small bugfix, small infra}`, NO risk keywords, no decomposition, no EVA arch-plan.
- **HIGH**: risk keywords (auth/payments/rls/credentials/migration/schema/security), `sd_type=feature`, orchestrator/decomposition, arch-plan, cross-repo.
- **MEDIUM**: everything else.
- **CONSERVATIVE-UP BIAS (load-bearing):** when uncertain, score UP one tier. Under-powering a complex SD (gate failures, rework, bad merges) costs more than the token savings — mirror the cheap-scout misclassification caveat.
- Compute deterministically where possible; for genuinely ambiguous SDs, a **Sonnet-scout triage pass** (Targeting Machine applied to triage). Store as `metadata.min_tier_rank` (the CHEAPEST sufficient rung on the ladder, see component 2) at creation/triage; dispatch + claim-eligibility then route to the cheapest LIVE worker rung ≥ that. Coarse complexity (trivial/standard/complex/critical) maps to the rung ordinal.

**2. Worker tier LADDER — CHAIRMAN-DECLARED, model×effort (refined 2026-06-26).** No worker-effort field exists in the fleet today (confirmed via grep); the chairman DECLARES each worker's rung manually, STAMPED on the session in `claude_sessions.metadata` (`model` + `effort`, surfaced as a single ordinal `tier_rank`). **TWO independent levers:** MODEL (Sonnet vs Opus = the big cost step) and EFFORT (low/med/high/max = the cheap reasoning dial). **Design principle (chairman): max the cheap dial, spend the expensive lever carefully** — so the cheapest rung is **Sonnet at MAX effort** (a capable-but-cheap floor, NOT a hobbled Sonnet-low), and the Opus rungs escalate by effort. The 4-rung ladder (cost/capability ascending), encoded as ordinal `tier_rank`:
  1. **Sonnet @ max effort** — cheapest capable; trivial/simple SDs
  2. **Opus @ low effort**
  3. **Opus @ medium effort**
  4. **Opus @ high effort** — most expensive; complex/risky SDs
Keep the ladder CARDINALITY as CONFIG (N rungs, not hardcoded 3/4). Per the coordinator, **PROTOTYPE 2 rungs first** (the extremes: Sonnet-max + Opus-high) to validate the token savings, then expand to the full 4 as worker capacity allows; degrade-gracefully holds for any live subset. **Model** = per-session Claude Code launch (`--model` / `/model` / a session-scoped env — NEVER the shared `.env`, which is global to all sessions); **effort** = the session reasoning-effort dial (the existing EFFORT budget concept). Auto-advertisement (worker self-tags at spawn) is a future nicety; the manual stamp unblocks v1 with near-zero new infra (read one metadata field).

**3. Assignment = intelligent, token-minimizing delegation (coordinator-owned).** The coordinator's explicit goal is to **assign the LOWEST sufficient effort to each task** (not burn high-effort tokens on simple work). Route an SD to the lowest-tier worker whose tier ≥ the SD's `effort_tier`. **Work-down, never up:** a higher-tier worker MAY claim below its tier when its own tier's queue is empty (so no worker idles while suitable work waits), but a lower-tier worker may NEVER claim above its tier. Coordinator owns this delegation logic.

**4. Worker self-restraint (the clean hook).** Add a rule to `classifyDispatchIneligibility(sdRow, ctx)` in `lib/fleet/claim-eligibility.cjs` returning `above_worker_tier` when `sdRow.effort_tier > ctx.workerTier`. The Low worker's self-claim/belt path then **skips anything above its tier** — exactly "the worker knows not to pick up complex work." The gate is already fail-open and `ctx`-aware; the rule slots in.

**5. Hard floors (safety).** Risk-keyword SDs are NEVER Low-eligible regardless of LOC (reuse classify-quick-fix force-escalate). Low worker = Sonnet floor. Conservative-up bias on ties.

**6. Re-tiering at LEO handoffs (worker-to-worker handoff) — chairman refinement.** Complexity is clearest at phase boundaries, so make the LEO handoffs (LEAD→PLAN→EXEC) the re-tiering checkpoints: at each handoff, **RE-SCORE `effort_tier` for the NEXT phase**; if it has escalated above the current worker's tier, **hand the SD off to a higher-tier worker** for that phase rather than the current worker proceeding under-powered (e.g. an SD that looked Low at LEAD but whose PRD reveals real complexity → EXEC runs on the High worker). **This is nearly free given database-first:** `sd_phase_handoffs` already holds full phase state, so a different-tier worker resumes the next phase seamlessly from the handoff record — no context loss. `handoff.js` is the natural hook (it already runs the gate pipeline + assesses the next phase; add a next-phase tier computation + a re-dispatch-if-escalated step). Keep a within-phase safety release for a rare mid-phase blow-up, but the handoff boundary is the primary, cleanest re-tiering point.

**7. Token-efficiency CALIBRATION LOOP (chairman refinement — DEFERRED follow-up B).** Makes the tiering SELF-OPTIMIZING instead of a static guess. Coordinator-owned closed loop. **MEASURE** per (tier_rank × sd_type): gate-pass-rate (LEO runs validation at every handoff), rework / validation-retry counts (the 2-retry rule, RCA triggers, re-opened SDs) — from the EXISTING gate pipeline + `sub_agent_execution_results` + retry data (no new telemetry needed). **TUNE bidirectionally:** when a task-class clears its rung cleanly (low rework, high gate-pass) → push that class DOWN toward the cheapest-sufficient rung (save tokens); when a rung over-fails on a class (high rework/gate-failure) → push UP. **CADENCE (chairman): INFREQUENT + SAMPLE-GATED** — only re-tune a task-class once there is a statistically meaningful sample of outcomes for it (never on 1–2 points); freeze-then-calibrate applied to tiering. Sequenced AFTER the CORE is running + producing outcome data. This is what turns the static rubric into "a worker-management system that maximizes token efficiency over time" — the coordinator continually evaluating worker performance and migrating work to the cheapest rung that still passes.

## Build vs reuse
- **REUSE:** classify-quick-fix rules, Work Item Routing tiers, the claim-eligibility gate, the dispatch path.
- **NEW:** (a) compute + persist `effort_tier` on SDs (a triage step; extend `leo-create-sd` / a triage rubric); (b) worker `effort_tier` advertisement at spawn; (c) the `above_worker_tier` claim-eligibility rule; (d) lowest-sufficient-tier dispatch matching; (e) mid-flight escalation.

## Risks / open questions (for the coordinator)
- Is `claim-eligibility.cjs` the right + only self-claim gate, or does dispatch also need a parallel rule?
- Does any worker-effort field already exist that the grep missed?
- Mis-scoring: is the conservative-up bias + mid-flight escalation sufficient, or do we need a re-tier feedback loop from gate-failure data?
- Throughput: with 3 single workers, does tier-locking risk a Low worker idle while High SDs queue (and vice versa)? Should higher-tier workers be allowed to "work down" (claim below their tier) when their tier's queue is empty, but never up? (Recommended: yes — higher tiers can claim down, never up.)

## Coordinator review — INCORPORATED (2026-06-26, fleet-dispatch owner)
Design assessed sound. Grounded findings + resolutions:
- **(a) TWO enforcement points, not one (correction).** `classifyDispatchIneligibility` is the shared SSOT for worker self-claim (PULL) AND the stale-session-sweep CLAIM_FIX — the `above_worker_tier` rule covers both. BUT **directed coordinator dispatch (WORK_ASSIGNMENT) BYPASSES claim-eligibility by design** (directed work is honored even when self-claim is off). So the tier must ALSO be enforced in the coordinator ASSIGNMENT logic (component 3). Build BOTH: claim-eligibility rule (pull/sweep) + assignment rule (directed).
- **(b) New primitive CONFIRMED.** grep = 0 worker-session effort field. The only effort routing is per-LLM-CALL (`model-routing.js getEffortForAgentAndPhase` + client-factory `EFFORT_TO_BUDGET` low/med/high). USEFUL: a worker's tier can MAP to that existing per-call effort budget — so `metadata.effort_tier` isn't just a label, it sets the actual token budget for that session's LLM calls (where the savings come from).
- **(c) Handoff re-tier (component 6) = feasible but MOST COMPLEX** — claim-transfer races, the target higher-tier worker may be BUSY (SD stalls between workers with no holder), re-entrancy. DEFER it; ship the core without it.
- **(d) LOAD-BEARING PREMISE TENSION:** the design needs ≥2–3 CONCURRENT workers, but ops currently run 1–2 (chairman keeps backing to 1 on session/usage TIME-LIMITS; at 1 right now). Cannot tier-lock a single worker. RESOLUTION: make **graceful-degradation-to-1 a design INVARIANT** — a lone worker takes ALL work regardless of tier; tiering activates only when ≥2 run. Residual: a High backlog with idle Low/Med bottlenecks on the High worker (work-down doesn't fix the reverse) — monitor; optional emergency tier-flex-up valve under sustained High backlog.
- **(e) SOURCE AS A PHASED ORCHESTRATOR:** Child-A CORE = rubric(1)+worker-field-read(2)+claim-rule(4)+floors(5) = minimal viable tier system; Child-B = assignment/directed-dispatch(3, coordinator's lane); Child-C (defer/last) = handoff-retier(6). **PROTOTYPE 2 tiers (Sonnet-floor / Opus-high) before 3** — validate the token-savings value before the full 3-tier complexity. CONST-002 clean: Adam designs, coordinator owns 3+4 (dispatch/claim-eligibility), chairman spawns + declares tiered workers.
