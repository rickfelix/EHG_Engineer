**Status:** Solomon-authored (Fable, 2026-07-10), propose-only (CONST-002). **Question answered:** *if the post-build band did not exist, what must launch/demand/operations/revenue/exit machinery do for a SOLO OPERATOR whose ventures must become real operating businesses at near-zero marginal chairman time?*
**Baselines that already exist (do NOT re-greenfield; audit as-built vs as-designed):** the demand/distribution engine design (2026-07-07: PLAN/EXECUTE/MEASURE split, Stage-21 artifact-only, 3/3 explicit skips, ZERO fake funnel gauges by design) and the deploy pipeline design (promote()/preview()/test-mode overlay). Chairman authority points non-negotiable throughout. Phase-1 process-proving posture applies (per the governed-posture requirement R2 of the Stage-0 spec — one posture SSOT governs both specs).

## O1 — Purpose: operating proof, not launch theater
This band converts a built-and-verified venture into a **real operating business**: real strangers arrive, convert, pay real money, get supported, and the system learns — at near-zero marginal chairman time. The band's product is **OPERATING PROOF** (traffic → conversion → revenue → retention → support, each observed on real machinery). A venture that "launches" but never runs this loop has not traversed the band, whatever its stage pointer says.

## O2 — Launch is a gate-crossing, not a ceremony
Launch = the deploy path's `promote()` (already built) + the chairman's S23/S24 product-review and kill gates (absolute, never auto) + **the Stage-0 thesis's pre-registered kill criteria armed as LIVE gauges from day one** (the R3 contract consumed here). No launch without: rollback tested, spend guardrails live, support intake reachable, and the demand plan's first channel actually executable (O9). Everything else — announcements, listing polish — is optional decoration, never a gate.

## O3 — Demand/distribution: the as-designed baseline governs
The greenfield for this half **already exists** (the 07-07 engine design); this spec incorporates it by reference and adds the audit posture: verify AS-BUILT honors AS-DESIGNED — the PLAN/EXECUTE/MEASURE split, artifact-only Stage-21, explicit 3/3 skips, and above all the **zero-fake-funnel-gauges principle**: a funnel gauge exists ONLY when real data flows through it. Channel selection is explicit and evidence-graded (which channel, why, expected CAC basis with a declared source); marketing/distribution templates (GTM docs, marketing copy, distribution setup) count as EXECUTION INPUTS only — **a document that drives no channel action is artifact theater by definition** and must be findable as such.

## O4 — Every operating metric is provenance-REACHED
No dashboard number without a runtime-reachable real source: visitors from real analytics, conversions from real events, revenue from the real payment rail (attribution rail→venture→channel — the shipped attribution work is the substrate), support metrics from real tickets, costs from real billing APIs. A gauge whose source is unreachable renders **"NO DATA"** — never a fabricated zero, a placeholder, or a demo value. This is the decorative-computation/gauge-vs-action class applied to operations, where it is deadliest: a solo operator GOVERNS BY these gauges; a fake gauge is a fake business.

## O5 — Operations = machine-run loops with escalation ladders
Each operating concern is a named, observable machine loop: **support** (intake → AI triage → respond → escalate-on-criteria), **incidents** (health probes → auto-remediation → escalate), **cost** (live spend watch → guardrail enforcement → escalate), **content/channel upkeep** (scheduled execution against the demand plan). Two invariants: (i) every loop's escalation criteria map to the only-the-chairman-can doctrine — everything else self-serves; (ii) **no loop may silently no-op** — each records last-ran-at + outcome so loop liveness is externally observable (the dormant-verifier class made structurally visible).

## O6 — Revenue is real-rail from day one
Payment machinery runs the REAL rail in test mode until launch; **live-mode flip is chairman authority** (spend/legal — only-the-chairman-can §7). Attribution must answer "which venture, which channel, how much, this week" as a REAL number — this is the income-replacement north-star gauge and inherits O4's provenance bar. Refund/failure/dispute handling is a machine loop (O5) with dollar-threshold escalation. Trivially small real revenue is a SUCCESS under Phase-1 posture: it proves the rail, attribution, and collection path end-to-end.

## O7 — Retention and the feedback loop close the circle
Real user feedback flows to the venture backlog (intake → dedupe → prioritize → recommend), churn/retention gauges are provenance-reached, and the loop's output is an **iterate-or-kill recommendation evaluated against the Stage-0 thesis's pre-registered kills** — the machine recommends, the chairman decides. A venture with users but no operating feedback loop is unfinished machinery, not a lifestyle business.

## O8 — Scale-or-exit decision machinery exists and is pre-scheduled
Post-launch reviews run on a pre-declared cadence: the machine assembles the evidence (gauges vs thesis vs posture) and RECOMMENDS scale / maintain / iterate / kill / exit; the chairman DECIDES. Under Phase-1 posture the review criterion is **process-proving** — "did the full loop genuinely run?" — not revenue thresholds. Exit (sale/shutdown/archive) is designed machinery with a checklist (data, customers, domains, billing teardown), not an improvisation; kill executes the Stage-0 pre-registered criteria without renegotiation.

## O9 — Capability-envelope honesty for channels and operations
The band declares what it can GENUINELY execute today: which acquisition channels the factory can actually run (organic/SEO needs content machinery; paid needs budget + account + creative; social needs accounts and cadence), which support/ops loops are actually wired. A demand plan or launch plan requiring an undeclared capability **fails at plan time** — it must be impossible for a plan step to reach EXECUTE and silently no-op (the Stage-0 R6 envelope, applied where violations bite hardest).

## O10 — Acceptance: the simulated full-life run
One synthetic venture traverses S20→26 on the REAL machinery — deploy test-mode overlay as containment (never mock beyond the enumerated divergences), APA-style instrumentation observing every loop fire: synthetic visitor → landing → conversion event → test-rail payment → attribution row → support ticket → triage → post-launch review recommendation. Three rules: (i) every O-requirement above maps to at least one observed behavior in the run; (ii) **"this loop cannot be driven even synthetically" is a first-class finding** (absence is the signal), never a routing-around; (iii) the run harness is built REUSABLE — it becomes the band's permanent regression instrument (the reality-simulation duty realized).
